# Health AI Apple 登录与离线优先同步改造计划

## Summary
- 保留现有 `users.id` 作为唯一主用户 ID，引入身份映射层，把设备登录、手机号登录、Apple 登录统一到同一用户体系。
- HealthKit 数据改为“先在 iPhone 本地采集并持久化排队”，不要求远程服务器在线；有网后优先上传到可达的局域网节点，再由现有服务端同步链路补到远端。
- 报告、计划、首页、数据页补齐本地持久化缓存，离线时优先展示最近成功数据与状态提示，避免白屏和整页失败。
- 完成后做服务重启、iPhone 真机安装和联调验证，签名先沿用当前工程配置。

## Key Changes
### 1. 统一用户 ID 与 Apple 登录
- 新增 `user_identity` 表，作为账号归一层，记录 `provider`、`provider_subject`、`user_id`、可选邮箱/声明信息；把现有 `users.phone_number`、`users.device_id` 回填成 identity 记录。
- 在 `users` 上增加合并状态字段，例如 `merged_into_user_id` 与禁用标记；被合并账号不硬删，保留为别名，避免多服务器场景下的删除同步问题。
- 新账号创建改成“按身份主体生成稳定用户 ID”：
  - 设备登录按 `device_id`
  - 手机号登录按 `phone_number`
  - Apple 登录按 Apple `sub`
  这样同一身份在 LAN/远端不同服务器首次登录时，不会各自生成不同用户 ID。
- 新增显式合并能力：当当前已登录账号去绑定 Apple，或验证手机号时发现该身份已落在别的账号上，就把当前账号设为目标账号，迁移所有用户域数据到目标账号，并把源账号标记为 merged。
- 新增 Apple 认证接口：
  - `POST /api/auth/apple/sign-in`
  - `POST /api/auth/apple/link`
- 服务端校验 Apple `identityToken`，保存 Apple 身份映射，返回与现有登录一致的 JWT/session 结构。
- `UserInfo` 与 `/api/auth/me` 扩展 provider 元数据，至少包含 `authProviders`、`email?`、Apple 绑定状态，供 iOS 渲染。
- iOS 端加入 Sign in with Apple capability/entitlement，在登录页增加 Apple 登录按钮，在设置页增加“绑定 Apple 账号”；当用户已登录时，Apple 流程直接走自动绑定当前账号。

### 2. HealthKit 本地优先与 LAN 优先补同步
- HealthKit 读取继续完全在手机端完成，不依赖任何服务器。
- 在移动端新增本地持久化同步队列，放到 `VitalCommandMobileCore`，使用 Application Support 下的 JSON 文件，不用 UserDefaults 存大对象。
- 队列按 `(kind, sampleTime)` 做去重合并；每次前台/冷启动同步时，先拉取 HealthKit 样本并写入本地队列，再尝试出队上传。
- 上传目标选择顺序固定为：
  1. 可达的局域网发现节点
  2. 当前配置的服务器
  3. 已保存的已知服务器 / 远端兜底地址
- 如果目标服务器没有当前 session，不同步 `auth_sessions`，而是由 App 用同一设备标识在目标服务器上做一次本地设备自动登录，再上传队列。
- 上传到局域网节点成功后，顺手调用现有服务端 peer sync；如果失败，依赖已有 30 秒调度器后续同步到远端。
- 现有 `/api/healthkit/sync` 不改成无鉴权接口，仍保持用户态写入；跨服务器连续性靠稳定身份映射与目标服务器本地 re-login 保证。

### 3. 离线缓存与无白屏
- 把缓存基础设施下沉到 `VitalCommandMobileCore`，统一提供 JSON cache store，便于 `swift test`。
- 增加持久化缓存对象：
  - dashboard
  - reports index
  - report detail
  - health plan dashboard
  - plan progress
  - import task list
  - 最近一次 HealthKit 本地采集/远端同步状态
- 首页沿用“缓存优先、网络刷新”模式，并把相同策略补到报告列表、报告详情、计划页、数据页。
- 各 ViewModel 在网络失败时保留已有缓存，不再直接切成失败空页；没有缓存时展示明确空状态，不出现白屏。
- 统一加离线/陈旧数据提示，展示缓存时间。
- 计划与报告模块这一版离线模式定为只读：
  - 可以看缓存
  - 接受建议、打卡、编辑计划等操作离线时禁用并给出提示
- DataHub 离线时显示待上传样本数、最近本地采集时间、最近成功同步结果，不阻塞页面加载。

### 4. 验证与交付
- 服务端补测试：
  - identity 回填
  - canonical user 解析
  - Apple token 校验成功/失败
  - Apple/手机号绑定触发的账号合并
  - 合并后 dashboard/report/plan 仍读到同一用户数据
  - HealthKit 同步仍保持幂等写入
- `VitalCommandMobileCore` 补测试：
  - Apple 登录请求/响应模型
  - 本地 cache 读写
  - 待同步队列去重与持久化
  - LAN 优先目标选择
- 回归验证顺序固定：
  - `npm test`
  - `npm run typecheck`
  - `swift test`
  - iOS 构建签名安装
  - 重启 Health 服务
  - 真机验证 Apple 绑定/登录、断网启动、断网查看缓存、恢复联网后补传、局域网到远端传播

## API / interface changes
- 新增接口：`POST /api/auth/apple/sign-in`、`POST /api/auth/apple/link`
- `UserInfo` / `/api/auth/me` 增加 provider 信息与可选邮箱
- 新增数据库结构：`user_identity`，以及 `users` 上的 merged/disabled 状态字段
- `user_identity` 纳入现有 server-to-server sync 表清单
- 新增移动端本地接口层：缓存存储、待同步队列、服务器优先级解析器

## Test scenarios
- 当前设备账号绑定 Apple 后，`users.id` 不变，历史报告/计划/指标全部继续可见。
- 当前设备账号与已有手机号账号发生融合时，数据迁移到当前账号，旧账号变为 merged alias。
- 完全离线时采集到的 HealthKit 样本会落到本地队列，App 重启后仍在。
- 远端不可达但局域网节点可达时，队列会先传 LAN，再由服务端同步到远端。
- dashboard、报告列表/详情、计划页离线时都能打开并显示缓存时间，不会白屏。
- 离线下计划页保持只读，联网恢复后再允许变更操作。

## Assumptions / defaults
- Apple 首次绑定策略采用“自动绑定当前已登录账号”。
- LAN 优先只用于 HealthKit 队列上传链路；普通页面读取仍以当前配置服务器为主，失败则落回缓存。
- 不做历史账号的启发式批量去重；只处理用户显式触发的当前账号融合。
- 离线期间不会在手机端本地重算 AI 报告或计划建议；新 HealthKit 数据要等服务器补同步后才体现在分析结果里。
- 真机安装先沿用仓库当前签名配置；若 provisioning 不匹配，再切到你的开发者组配置。
