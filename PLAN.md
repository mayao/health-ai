# Health 第二轮洞察与饮食扩展方案

## Summary
本轮按两个目标一起做：

1. 把 `体检分析 / 基因洞察` 从“长文本卡片”升级成更结构化、更易读的洞察界面，重点解决 `AI 综合分析` 留白过大、`体检洞察总览` 文字过密的问题。  
2. 打通“基因图片上传 + 饮食图片上传 + 首页/计划/报告联动”的完整数据链，其中饮食模块按你确认的规则实现为：
   - 只保留**每日汇总**
   - 同一天多次上传**按餐次累加**

## Key Changes

### 1. 洞察页结构化重构
- 重做体检/基因洞察顶部总览卡：从单段长摘要改为“结论标题 + 2-3 条关键点 + 严重度统计条 + 生成时间”的紧凑结构，移除当前右上角标签造成的大块留白。
- `AI 综合分析` 不再单独占据空白区域，改成嵌入式摘要条，与总览内容合并呈现。
- 洞察项卡片改成“严重度标签 + 类型标签 + 标题 + 解释 + 建议动作 + 关联指标”六段式结构，减少连续大段正文。
- `体检洞察总览` 和 `基因洞察总览` 都按严重度分组展示，同时每条洞察补一个轻量分类标签，避免用户只能靠长文判断主题。
- 字体和排版统一调整为更疏朗的层级：标题用 `headline/title3`，解释正文拆段，建议动作单独高亮块，指标 chip 保持低视觉噪音。

### 2. 基因上传改造成正式能力
- 数据上传页放开基因的 `拍照 / 相册 / 文件导入`，不再只允许固定文件格式。
- 后端新增**专用基因导入解析链**，不再复用当前只会写 `metric_record` 的文档指标导入逻辑。
- 图片/PDF 基因报告：
  - iOS 继续复用现有本地 OCR
  - 服务端将 OCR 文本送入基因解析器，产出结构化 finding 后写入 `genetic_findings`
- CSV/JSON/XLSX 基因报告：
  - 走同一个基因标准化入口，统一落成 `trait_code / risk_level / evidence_level / summary / suggestion`
- 已知 trait 映射到规范 `trait_code`；无法识别的 trait 也保留为用户私有 finding，不丢弃，只是不参与部分跨模块联动增强。
- 导入成功后刷新首页基因模块、基因洞察缓存和相关报告数据。
- 全链路保持严格 `user_id` 隔离，切换账号后只读取当前账号的 `genetic_findings` 与关联 source。

### 3. 数据上传页 UI 重做
- `添加数据` 卡片改成更清晰的“图标 + 标题 + 单行价值说明 + 选中态”结构，移除现在堆叠的格式 chips。
- `拍照 / PDF / CSV` 等说明不再在卡片层面反复展示，改为用户选中类型后在操作面板里出现。
- 基因、饮食、体检等卡片统一视觉密度，避免图标、标题、副标题同时挤在一行内。
- 新增 `饮食健康` 上传类型，和现有类型并列。

### 4. 饮食健康模块 MVP
- 新增 `diet` 导入类型，支持从饮食/进食图片上传。
- 饮食图片不走 OCR 指标抽取，而是走**多模态食物识别服务**：
  - 输入：图片文件
  - 输出：食物类型列表、估算总热量、上传日期
- 存储策略按本轮确认的范围执行：
  - 不建长期餐次历史页
  - 不保留餐次级产品能力
  - 只把当天聚合结果写入统一指标
- 同日多次上传按餐次累加，写入至少两个 diet 指标：
  - `diet.calories_intake_kcal`
  - `diet.meal_upload_count`
- 食物识别明细仅保留在导入任务原始 payload / 日志中，用于上传结果展示和排障，不作为首页长期历史结构。
- 若当前环境没有可用的图像分析模型，饮食上传任务要明确失败并返回可读错误，而不是静默成功。

### 5. 首页 / 计划 / 报告联动饮食维度
- 首页新增饮食模块：
  - 有数据时展示当天/最近一次热量、近 7 天平均、上传覆盖情况
  - 无数据时展示弱化空态，并给出上传入口 CTA
- `HealthHomePageData` 的趋势图增加 diet 图，首页展示最近 30 天热量趋势。
- `sourceDimensions` / `dimensionAnalyses` 增加饮食维度，使首页整体分析把饮食与体重、血脂、运动一起看。
- 计划模块增加 diet 建议：
  - 有饮食数据时，基于最近热量趋势、记录覆盖率、与体重/体脂联动生成建议
  - 无饮食数据时，优先给“先连续上传 3-7 天饮食照片”的采集型建议
- 报告模块增加饮食反馈：
  - 周/月摘要纳入饮食趋势与记录连续性
  - 结构化 insight 增加饮食相关规则，如“热量记录不足”“近期热量上行且体重同步上行”“热量回落并伴随体重改善”等
- 报告摘要生成同时切到现有 provider priority 体系，避免首页/报告的模型选择逻辑继续分叉。

## Public APIs / Types
- `ImporterKey` 新增 `diet`
- `UnifiedMetricCategory` 新增 `diet`
- `HealthCharts` 新增 `diet`
- `HealthImportOption` 新增 diet 选项，并把 genetic 的格式说明放开到图片/文档/结构化文件
- `InsightItem` 新增可选分类字段，如 `categoryLabel`
- `DocumentInsightResponse` 新增结构化摘要字段，建议为：
  - `summaryHeadline`
  - `summaryHighlights: string[]`
  - 保留原 `summary` 作为兼容字段
- 服务端新增 diet 指标定义，至少包含：
  - `diet.calories_intake_kcal`
  - `diet.meal_upload_count`

## Test Plan
- 基因上传：
  - 图片/PDF/CSV/JSON/XLSX 均能创建任务并正确落库到 `genetic_findings`
  - 账号 A 上传后，账号 B 首页、基因洞察、报告均不可见
  - 上传后切换账号再切回，缓存不会串用户
- 饮食上传：
  - 单张图片可识别并写入当天热量
  - 同一天连续上传 2-3 次会累加到当天总摄入
  - 无可用视觉模型时返回明确失败原因
- 首页联动：
  - 有饮食数据时展示热量模块和趋势
  - 无饮食数据时展示弱化空态和上传引导
  - 饮食、体重、血脂联动文案只基于当前用户数据生成
- 计划/报告：
  - 有 diet 数据时出现 diet 建议与反馈
  - 无 diet 数据时出现采集引导而不是伪造反馈
- 洞察 UI：
  - 顶部无异常留白
  - 长文本被拆成结构化块
  - 小屏与较大动态字体下无拥挤、截断和明显视觉失衡

## Assumptions
- 本轮饮食模块只做**每日汇总**，不做餐次历史页和餐次编辑。
- 同日多次饮食上传按“新的进食记录”处理，热量**累加**到当天。
- 饮食首页与报告以“趋势反馈”和“记录覆盖”作为主判断，不在本轮引入个性化精确热量处方。
- 基因未知 trait 允许保存为用户私有 finding，避免因平台差异直接丢数据。
- 本计划只定义实现方案；真正的服务重启、本地/远程部署、手机安装和 TestFlight 上传在退出 Plan Mode 后按现有流程执行。
