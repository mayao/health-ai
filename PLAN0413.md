# Health AI 面向 TestFlight 普通用户的发布优化方案

## Summary
- 目标版本按 `TestFlight 内测` 标准收口，优先解决 4 类问题：开发能力暴露、使用门槛高、Apple 登录不稳定、缺少正式协议与隐私承诺。
- 登录策略固定为：普通用户首启先看引导，再明确选择 `Apple 登录` 或 `本机快速进入`；不再静默自动创建设备账号。
- 账号切换只保留给 `user-self`；其他调试能力保留，但统一折叠到“高级设置”。
- 隐私合规先采用 `邮件申请导出/删除`，联系人固定为 `Health AI团队 / yao.ma@qq.com`；不再把占位式隐私能力直接暴露成“可用功能”。

## Key Changes
### 1. 账号与开发能力收口
- 将“账号切换”权限从“设备账号可切换”改成“仅 `user-self` 可切换”，服务端和客户端同时收口，避免只做前端隐藏。
- `创建测试账号` 与账号切换归为同一组开发能力，仅 `user-self` 可见。
- 设置页中的 `AI 模型选择 / 服务地址 / 快速切换服务器 / 局域网发现` 保留，但改为默认折叠的“高级设置”；普通用户不再第一屏看到这些内容。
- 登录页移除显性开发元素：默认不展示服务器地址、设备标识、显式“切换服务器”；保留隐藏诊断入口作为内部调试手段。
- 关键接口/类型改动：
  - 在 `/api/auth/me` 返回的 `UserInfo` 中新增服务端下发的能力字段，建议使用 `capabilities`，至少包含：`canSwitchAccounts`、`canCreateTestUsers`、`canSeeAdvancedSettings`、`canUseDirectDeviceEntry`。
  - `/api/auth/users` 的 `canSwitchUser` 与服务端 `canUserSwitch` 逻辑统一改为仅 `user-self` 为 `true`。
  - iOS 端不再用 `user.id == "user-self"` 到处硬编码 UI 判断，而是优先消费 `capabilities`；`user-self` 仅作为服务端判定来源。

### 2. 普通用户可理解的首启与说明体系
- 新增 `首次启动引导卡片`，放在登录前，内容固定为 4 屏：
  - Health AI 是做什么的：看趋势、做整理、给行动建议，不替代医生诊断。
  - 首页与 AI 洞察怎么看：重点说明“结论是趋势解释，不是医疗诊断”。
  - 趋势 / 报告 / 数据上传分别适合什么场景。
  - 隐私与数据控制：支持 Apple 健康同步、手动上传、邮件申请导出或删除。
- 登录页 CTA 重写：
  - Apple 登录保持主按钮。
  - “跳过验证，直接进入”改成更友好的明确文案，例如“使用本机快速进入”。
  - Apple 失败时给出用户能理解的文案和下一步，不直接暴露裸状态码。
- 设置页把当前 4 行“使用说明”替换成一个可复用的说明入口页，结构与首启引导一致，但更完整，便于回看。
- 同步清理明显开发语气或会削弱信任的文案：如“种子数据”“切换服务器地址”“占位返回”“设备标识”等。

### 3. Apple 登录 502 稳定性修复
- 现状链路是：iOS 获取 Apple credential -> 服务端 `/api/auth/apple/sign-in` -> 服务端实时拉取 Apple JWKS 并校验 token。当前路由把所有异常都按 `401` 处理，且上游网络异常容易演变成公网 `502` 体验。
- 服务端改造方案：
  - 在 Apple 公钥拉取上增加明确超时控制和分类错误。
  - 把 Apple 公钥缓存改成“可复用旧缓存”的策略：命中已缓存 key 时，即使当次拉取失败也优先用缓存校验。
  - 对 `Apple 上游不可达 / 公钥获取失败 / 超时` 返回 `503` 级别的稳定错误，而不是伪装成 `401`。
  - 对 `identity token 无效 / audience 不匹配 / 签名校验失败` 保持 `401`。
  - 记录结构化日志，至少包含：`route`、`error kind`、`audience`、`kid`、`是否命中缓存`、`hostname`，便于区分主站/备站问题。
- 客户端改造方案：
  - 对 `5xx` Apple 登录失败展示统一提示，例如“Apple 登录服务暂时不可用，请稍后重试或先使用本机快速进入”。
  - 对 `401` 类 Apple 授权失败展示“授权已失效/返回无效，请重新尝试”。
  - 登录页不显示“状态码 502”，只保留用户可执行动作：重试、返回、改用本机快速进入。
- 验收目标：
  - 主站与备站都能稳定完成 Apple 登录。
  - Apple 上游偶发失败时，用户看到的是可理解提示，不是裸错误码。
  - 日志可以区分是 token 本身问题，还是 Apple 公钥拉取问题。

### 4. 用户协议、隐私政策与隐私入口
- 法务展示策略固定为：`设置页常驻入口`，不在登录前强制勾选。
- 由于 App Store 需要公开的隐私政策 URL，本轮同时新增公开网页：
  - `/legal/privacy`
  - `/legal/terms`
- iOS 设置页增加独立入口：
  - `用户协议`
  - `隐私政策`
  - `隐私与数据申请`
- 协议文案基线：
  - 主体：`Health AI团队`
  - 联系邮箱：`yao.ma@qq.com`
  - 语气：清晰、克制、面向普通用户，不使用“平台保留最终解释权”这类强对抗表达。
- `用户协议` 需要覆盖：
  - 服务定位与非医疗诊断声明
  - 账号使用方式（Apple 登录 / 本机快速进入）
  - 用户上传和同步数据的授权边界
  - 使用限制与用户责任
  - 服务中断、版本变更、联系方式
- `隐私政策` 需要覆盖：
  - 收集的数据类型：账号标识、健康与健身数据、上传文件、设备基础诊断信息
  - 使用目的：登录识别、趋势分析、报告生成、AI 洞察、同步与问题排查
  - HealthKit/Apple 健康数据说明
  - 数据存储与保护承诺
  - 不出售个人信息、不将健康数据用于广告定向
  - 数据访问、更正、导出、删除的申请方式：通过 `yao.ma@qq.com`
  - 生效日期与版本更新方式
- 数据页现有“导出数据 / 删除数据”占位按钮改为“隐私与数据管理”说明卡，不再调用占位接口；统一引导用户发邮件申请。
- App Store 发布侧同步补充：
  - 隐私政策 URL
  - Support URL
  - App Privacy / Nutrition Label 声明，按实际采集数据填写

### 5. 面向普通用户还需要同步调整的方向
- 发布前清理所有“未完成/占位”观感：
  - 占位式隐私导出删除
  - 开发说明式文案
  - 与普通用户无关的错误细节
- 做一次“首屏可信度”收口：
  - 登录页、设置页、数据页只保留普通用户能理解的功能和词汇。
  - 全局保留统一的非医疗诊断声明。
- 做一次“数据边界”收口：
  - 删除或改写会让用户误以为能看到他人/测试账号数据的提示。
  - 数据页“隔离说明”去掉“切换账号后”一类内部测试语境，改成“当前仅展示本账号已授权或已上传的数据”。
- 做一次“发布元信息”收口：
  - App Store 文案、截图、描述、隐私标签、支持邮箱、隐私政策 URL 一致。
  - 对外描述不要出现 `mock / demo / seed / local-first 原型 / 占位`。
- 做一次“稳定性与可观测性”收口：
  - Apple 登录、首页加载、数据上传、HealthKit 同步至少有服务端错误分类日志。
  - 关键失败场景都提供用户可执行的下一步，而不是技术错误原文。

## Test Plan
- 权限与可见性
  - `user-self` 登录时可看到账号切换和创建测试账号。
  - 普通账号登录时看不到账号切换相关入口。
  - 普通账号仍可看到“高级设置”，但默认折叠。
- 首启与登录
  - 全新安装首次打开时先看到引导，不自动进入设备账号。
  - 用户可显式选择 `Apple 登录` 或 `本机快速进入`。
  - Apple 登录成功后，账号信息正确落盘并可复启恢复。
  - Apple 登录遇到无效 token、上游失败、超时三类错误时，提示文案和状态分流正确。
- 法务与隐私
  - 设置页能打开 `用户协议`、`隐私政策`、`隐私与数据申请`。
  - `/legal/privacy` 与 `/legal/terms` 可公开访问，适合 App Store 填写 URL。
  - 数据页不再出现“导出占位返回/删除占位返回”。
- 文案与发布体验
  - 登录页、设置页、数据页不出现“种子数据 / 状态码 502 / 切换服务器地址 / 设备标识 / 占位返回”等普通用户不该看到的表达。
  - 普通用户能通过引导和说明页理解首页、AI 洞察、趋势、报告、数据上传各自作用。
- 环境验证
  - 主站与备站分别完成 Apple 登录联调。
  - 在 Apple JWKS 拉取失败的模拟场景下，服务端返回稳定 503，客户端提示友好。
  - App Store Connect 中的隐私政策 URL、Support URL、App Privacy 声明与应用实际行为一致。

## Assumptions
- 本轮不做真正的自助导出/删除能力，只改为邮件申请流程。
- 协议与隐私文案先按 `中文` 单语版本落地。
- 若未另行指定，默认以 `18+ 或在监护人指导下使用` 的保守表述处理未成年人条款，并在正式上架前做一次法务复核。
- 公开文案结构参考以下公开要求与同行做法：
  - Apple 要求 iOS App 提供公开可访问的隐私政策 URL，并在 App Store Connect 准确声明数据采集做法：[Manage app privacy](https://developer.apple.com/help/app-store-connect/manage-app-information/manage-app-privacy)
  - Apple 要求在产品页提供隐私链接，并说明数据收集与数据选择入口：[App Privacy Details](https://developer.apple.com/app-store/app-privacy-details/)
  - Apple 强调提交新版本前需准确披露隐私与数据使用信息：[User Privacy and Data Use](https://developer.apple.com/app-store/user-privacy-and-data-use/)
  - 薄荷健康在 App Store 描述中同时提供免责声明、HealthKit 说明、隐私政策、用户协议和联系方式，可作为健康类 App 的展示结构参考：[薄荷健康 App Store 页面](https://apps.apple.com/us/app/%E8%96%84%E8%8D%B7%E5%81%A5%E5%BA%B7-ai%E5%87%8F%E8%82%A5%E5%81%A5%E8%BA%AB%E8%BD%BB%E6%96%AD%E9%A3%9F%E4%BD%93%E9%87%8D%E7%AE%A1%E7%90%86%E5%B9%B3%E5%8F%B0/id457856023?l=zh-Hans-CN)
  - MyFitnessPal 把访问、更正、删除、申诉等数据主体权利写得很明确，适合参考“权利说明”结构：[MyFitnessPal Privacy Rights](https://support.myfitnesspal.com/hc/en-us/articles/34887801585549-What-are-my-Privacy-Rights)
  - 华为运动健康提供停止服务、删除云端数据、撤销授权等路径说明，适合参考“隐私管理入口”结构：[华为运动健康隐私声明](https://privacy-dre.dbankcdn.com/agreement/scope/huawei/health/privacy-statement/cn_b0_20250627_0/default/privacy-statement-zh-cn.htm?bgmode=white&ctype=huawei&timestamp=2025100117)
